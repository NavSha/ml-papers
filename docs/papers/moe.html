Switch Transformers: Scaling to Trillion Parameter Models with Simple and Efficient Sparsity

Fedus et al., Google Brain · JMLR 2022 · arXiv 2101.03961

Google scaled a model toward a trillion parameters while compute per token stayed flat — a learned router sends each token to exactly one expert, and pretraining ran up to 7× faster on the same budget.

the aha · the routing diagram, redrawn

One token, one expert

You have drawn this diagram before — it is a load balancer. The Switch layer replaces each dense feed-forward block with many parallel expert FFNs and a small learned router that dispatches each token to exactly one of them; the rest sit idle. Total parameters can scale toward 1T while FLOPs per token stay constant — and pre-training ran up to 7× faster than T5-Base at the same computational budget. A hospital triage desk works the same way: each patient sees one appropriate specialist instead of every doctor on staff, so the institution's total knowledge can grow 100× without any single visit taking longer.

step 01 / 04

The dense tax

In a normal Transformer every token flows through every parameter. Double the parameters and you double the cost of every single token. That coupling is what made scaling expensive — until someone broke it.

step 02 / 04

One expert per token

Replace the dense layer with many parallel experts and a small router. Each token is sent to exactly one expert; the rest sit idle. Total parameters can climb into the trillions while the compute spent on any given token stays flat.

step 03 / 04

The receipts

Switch Transformers reported up to a 7× pre-training speedup over the dense T5-Base baseline at the same compute budget. Same money, far more model — the router is just a load balancer, and every engineer has drawn this exact box-and-arrows picture for microservices.

step 04 / 04

The pricing lesson

This is why 'parameters' and 'inference cost' stopped being the same axis. When you compare models on price and speed, ask about ACTIVE parameters, not the headline size — an MoE with hundreds of billions of total params can cost like a small dense model per token. It's a big part of why API prices keep falling while capability rises.

what it changed

Ask about active parameters

"Parameters" and "inference cost" are no longer the same axis. An MoE model with hundreds of billions of total parameters can cost like a small dense model per token — the economic trick behind most frontier models (Mixtral, DeepSeek-V3, and reportedly GPT-4). So when you compare models on price, speed, and capability, ask about active parameters, not headline size. This decoupling is a large part of why API prices keep falling while capability rises — and why open MoE releases undercut closed dense pricing.

honest limits

Experts, not specialists

The misconception: "experts are human-legible domain specialists — one does code, one does law." Routing is per-token and learned under load-balancing pressure, so the specialization that emerges is mostly token-level patterns — punctuation, syntax, language — not subjects. And MoE is not an ensemble of independent models: the experts are interleaved inside every layer of one network.

The real cost is a new operations problem: the router has to keep load balanced so one expert isn't slammed while the others sit idle — the triage desk only works while no single doctor's queue backs up.
